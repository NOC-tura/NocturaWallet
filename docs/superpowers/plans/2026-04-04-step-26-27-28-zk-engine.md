> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the complete fee engine (transparent SOL + shielded NOC, pre-TGE/post-TGE, staking discounts), Merkle tree incremental sync module (light client, depth 32), and ZK prover module (hosted primary + local stub + crash-safe queue). Per architecture validation: hosted prover uses real HTTP calls, local Polygen is stubbed, proof queue persists in MMKV.

**Architecture:** FeeEngine combines transparent fees (SOL markup) and shielded fees (NOC, TGE-gated, discount-aware). MerkleModule syncs incrementally from the API with local root verification. ZKProverModule implements the hosted→local→queue fallback chain with real HTTP for hosted, stub for local, and persistent MMKV queue.

**Tech Stack:** @noble/hashes (Poseidon field prep), pinnedFetch (API calls), MMKV (queue + merkle state), BigInt

2. getPendingEntries returns pending entries
3. updateEntry changes status
4. Entries >10min marked as failed

### ZK Prover Tests (6):
1. generateDepositProof tries hosted first (mock pinnedFetch)
2. Falls back to queue on hosted failure
3. getProverMode returns 'hosted'
4. Local prover returns { supported: false }
5. Witness arrays are zeroized after proof generation
6. NEVER sends sk_spend to hosted prover (verify params don't contain 'spend')

### Implementation:
- `ProofQueue`: MMKV-persistent, CRUD on ProofQueueEntry, auto-fail >10min entries
- `LocalProver`: stub with `checkViability() → {supported: false}`
- `ZKProverManager`:
  - `generateDepositProof/TransferProof/WithdrawProof`: hosted → local → queue
  - Hosted: POST /v1/prove/{circuit_type} via pinnedFetch, 8s timeout
  - Local: localProver.checkViability() → skip if not supported
  - Queue: addToQueue for retry
  - Witness zeroization after proof generation
  - NEVER includes sk_spend in hosted request params
